# ADR-037: Dev/Prod Docker Convergence on FrankenPHP

- **Status:** Accepted
- **Date:** 2026-06-01
- **Depends on:** ADR-019 (Deployment Infrastructure)
- **Related:** ADR-036 (manual OSM refresh)

## Context

Development and production ran two divergent Docker topologies:

- **Dev:** `caddy:2-alpine` (proxy) + `dunglas/mercure` (hub) + `php` running the Symfony CLI server, plus `mailcatcher`/`ollama`. Defined in a self-contained `compose.yaml`.
- **Prod:** a single FrankenPHP image (`php`) embedding the reverse proxy, the Mercure hub and the PHP app. Defined in a self-contained `compose.prod.yaml`.

The two files duplicated most service definitions and drifted: a runbook editing `compose.prod.yaml` did not match what a developer saw locally, and the dev `caddy`/`mercure` services behaved differently from the prod FrankenPHP edge. The Dockerfiles already shipped a `frankenphp_dev` stage and a pwa `dev` target, so the divergence was incidental, not required.

## Decision

Run **the same FrankenPHP topology in dev and prod**, with a layered Compose setup:

- **`compose.yaml`** — the iso-prod base and single source of truth: FrankenPHP `php` edge (proxy + embedded Mercure + app), `worker`, `pwa`, `database`, `redis`, `valhalla`, `provisioner`; `read_only`, Docker secrets, `deploy`/replicas, prod env. Read as-is by CI and Coolify.
- **`compose.dev.yaml`** — dev overrides layered on top: `frankenphp_dev` / pwa `dev` build targets, source bind-mounts, Xdebug, `read_only: false`, lexik filesystem JWT keys (no Docker secrets), the Mercure `demo` UI, plus the dev-only `mailcatcher` and `ollama` services.

`compose.prod.yaml` and the standalone `caddy` / `mercure` services are removed.

### Invocation

- **Dev:** `docker compose -f compose.yaml -f compose.dev.yaml …`, wired through the Makefile via `export COMPOSE_FILE=compose.yaml:compose.dev.yaml`. `make start-dev` adds `COMPOSE_PROFILES=routing` to bring up Valhalla as before.
- **Prod:** `docker compose -f compose.yaml …` (CI sets `COMPOSE_FILE: compose.yaml`; Coolify points its "Docker Compose Location" at `compose.yaml`).

### Why `compose.dev.yaml` and not `compose.override.yaml`

Docker auto-loads `compose.override.yaml` next to `compose.yaml`. Coolify's exact `docker compose` invocation for a deployed app is not documented, and that auto-load is a known footgun (dev settings silently applied in prod). Naming the overrides `compose.dev.yaml` (not auto-loaded) guarantees prod reads `compose.yaml` alone, regardless of how Coolify invokes Compose. `compose.override.yaml` stays free (and gitignored) for per-developer local tweaks.

## Consequences

### Positive

- One topology to reason about; dev mirrors prod (FrankenPHP edge, embedded Mercure).
- No duplicated service definitions; the override carries only deltas.
- Fewer dev services (no standalone `caddy` / `mercure`).

### Negative

- Dev installs Composer deps and runs migrations at first boot (vendor is not baked into `frankenphp_dev`), so the first `make start-dev` is slower; the dev `php` healthcheck uses a longer `start_period` to absorb it.
- A few keys need the Compose `!override` / `!reset` tags so the dev layer replaces (rather than merges into) the prod volume and secret lists.

### Neutral

- JWT differs by environment by design: prod uses Docker secrets (`/run/secrets`), dev uses lexik filesystem keys generated by the entrypoint (`config/jwt`).
